# 配置管理热更新接口文档

## 1. 接口概述

本文档描述了配置管理中心（config_manager.html）所需的后端API接口，用于支持配置的获取、更新和恢复默认值等操作。这些接口旨在实现配置的热更新功能，确保配置变更能够实时生效，无需重启服务。

## 2. 接口设计

### 2.1 获取配置列表接口

**接口功能**：获取所有配置项的详细信息，包括当前值、默认值、类型、描述等。

**请求信息**：
- **URL**：`/api/configs`
- **方法**：GET
- **请求参数**：无

**响应信息**：
- **状态码**：200 OK
- **响应格式**：JSON
- **响应数据结构**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": [
      {
        "id": 1,
        "key": "api.timeout",
        "group": "API",
        "value": "30000",
        "defaultValue": "30000",
        "type": "number",
        "description": "API请求超时时间（毫秒）",
        "operator": "admin"
      },
      {
        "id": 2,
        "key": "database.connection.max",
        "group": "数据库",
        "value": "10",
        "defaultValue": "10",
        "type": "number",
        "description": "数据库最大连接数",
        "operator": "admin"
      }
    ]
  }
  ```

### 2.2 应用配置变更接口

**接口功能**：将前端修改的配置项应用到后端并热更新生效。

**请求信息**：
- **URL**：`/api/configs/apply`
- **方法**：POST
- **请求参数**：
  ```json
  {
    "configs": [
      {
        "key": "api.timeout",
        "value": "60000"
      },
      {
        "key": "cache.ttl",
        "value": "7200"
      }
    ],
    "operator": "admin"
  }
  ```

**响应信息**：
- **状态码**：200 OK
- **响应格式**：JSON
- **响应数据结构**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "updatedCount": 2,
      "lastUpdateTime": "2026-03-12 15:30:45"
    }
  }
  ```

### 2.3 恢复默认配置接口

**接口功能**：将所有配置项恢复到默认值并热更新生效。

**请求信息**：
- **URL**：`/api/configs/restore-defaults`
- **方法**：POST
- **请求参数**：
  ```json
  {
    "operator": "admin"
  }
  ```

**响应信息**：
- **状态码**：200 OK
- **响应格式**：JSON
- **响应数据结构**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "restoredCount": 8,
      "lastUpdateTime": "2026-03-12 15:35:20"
    }
  }
  ```

### 2.4 获取单个配置详情接口

**接口功能**：获取单个配置项的详细信息，用于编辑操作。

**请求信息**：
- **URL**：`/api/configs/{key}`
- **方法**：GET
- **请求参数**：路径参数 `key`（配置键名）

**响应信息**：
- **状态码**：200 OK
- **响应格式**：JSON
- **响应数据结构**：
  ```json
  {
    "code": 200,
    "message": "success",
    "data": {
      "id": 1,
      "key": "api.timeout",
      "group": "API",
      "value": "30000",
      "defaultValue": "30000",
      "type": "number",
      "description": "API请求超时时间（毫秒）",
      "operator": "admin"
    }
  }
  ```

## 4. 业务流程

### 4.1 配置管理流程

1. **加载配置**：前端调用 `/api/configs` 接口获取所有配置项，渲染到配置列表页面。
2. **编辑配置**：用户点击编辑按钮，前端调用 `/api/configs/{key}` 接口获取单个配置详情，显示编辑模态框。
3. **暂存变更**：用户修改配置值后点击保存，前端在本地标记配置为已修改状态。
4. **应用变更**：用户点击"应用所有变更"按钮，前端调用 `/api/configs/apply` 接口，将所有已修改的配置发送到后端。
5. **热更新**：后端接收配置变更，更新配置并热加载生效，返回更新结果。
6. **恢复默认**：用户点击"恢复默认配置"按钮，前端调用 `/api/configs/restore-defaults` 接口，后端将所有配置恢复到默认值并热加载生效。

### 4.2 状态管理

- **未修改状态**：配置值与默认值相同，显示为灰色状态条。
- **已修改状态**：配置值与默认值不同，显示为橙色状态条，等待应用。
- **应用中状态**：配置正在热更新到后端，显示加载动画。
- **已应用状态**：配置已成功热更新到后端，显示成功提示。

## 5. 错误处理

| 错误码 | 错误信息 | 描述 |
|-------|---------|------|
| 400 | Invalid configuration format | 配置格式无效 |
| 401 | Unauthorized | 未授权访问 |
| 403 | Forbidden | 禁止访问 |
| 404 | Configuration not found | 配置项不存在 |
| 500 | Internal server error | 服务器内部错误 |
| 503 | Service unavailable | 服务不可用 |

## 6. 安全考虑

1. **权限控制**：配置管理接口应进行严格的权限控制，只有管理员或具有配置管理权限的用户才能访问。
2. **数据验证**：后端应对配置值进行类型和范围验证，确保配置值的合法性。
3. **审计日志**：记录所有配置变更操作，包括操作人、操作时间、变更内容等。
4. **防止注入**：对配置键名和值进行安全过滤，防止注入攻击。
5. **HTTPS传输**：所有接口应使用HTTPS协议传输，确保数据安全。

## 7. 性能优化

1. **缓存机制**：后端可对配置数据进行缓存，减少数据库查询。
2. **批量操作**：支持批量应用配置变更，减少网络请求次数。
3. **异步处理**：热更新操作可采用异步方式处理，避免阻塞请求。
4. **增量更新**：只传输和处理已变更的配置项，减少数据传输量。
5. **监控告警**：对配置变更和热更新过程进行监控，及时发现和处理异常。

## 8. 前端实现参考

### 8.1 核心功能实现

```javascript
// 获取配置列表
async function fetchConfigs() {
  try {
    const response = await fetch('/api/configs');
    const data = await response.json();
    if (data.code === 200) {
      configData = data.data;
      renderConfigList();
      updateConfigStats();
    }
  } catch (error) {
    showNotification('获取配置失败', 'error');
  }
}

// 应用所有变更
async function applyAllChanges() {
  const changedConfigs = configData.filter(c => c.isChanged);
  if (changedConfigs.length === 0) {
    showNotification('没有待应用的变更', 'info');
    return;
  }
  
  const configsToApply = changedConfigs.map(c => ({
    key: c.key,
    value: c.value
  }));
  
  try {
    const response = await fetch('/api/configs/apply', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        configs: configsToApply,
        operator: "当前用户"
      })
    });
    
    const data = await response.json();
    if (data.code === 200) {
      configData.forEach(config => {
        config.isChanged = false;
      });
      renderConfigList();
      updateConfigStats();
      document.getElementById('lastUpdateTime').textContent = data.data.lastUpdateTime;
      showNotification(`已成功保存 ${data.data.updatedCount} 项配置变更并热更新生效`, 'success');
    }
  } catch (error) {
    showNotification('应用变更失败', 'error');
  }
}

// 恢复默认配置
async function confirmRestoreDefaults() {
  try {
    const response = await fetch('/api/configs/restore-defaults', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        operator: "当前用户"
      })
    });
    
    const data = await response.json();
    if (data.code === 200) {
      // 重新获取配置列表
      await fetchConfigs();
      document.getElementById('lastUpdateTime').textContent = data.data.lastUpdateTime;
      showNotification('已成功恢复所有默认配置到后端并热更新', 'success');
    }
  } catch (error) {
    showNotification('恢复默认配置失败', 'error');
  }
  closeConfirmModal();
}
```

## 9. 后端实现参考

### 9.1 配置存储设计

推荐使用数据库存储配置信息，表结构设计如下：

**configs表**
| 字段名 | 数据类型 | 约束 | 描述 |
|-------|---------|------|------|
| id | INT | PRIMARY KEY AUTO_INCREMENT | 配置ID |
| key | VARCHAR(255) | UNIQUE NOT NULL | 配置键名 |
| group | VARCHAR(100) | NOT NULL | 配置分组 |
| value | TEXT | NOT NULL | 当前值 |
| default_value | TEXT | NOT NULL | 默认值 |
| type | VARCHAR(20) | NOT NULL | 配置类型 |
| description | TEXT | | 配置描述 |
| operator | VARCHAR(100) | | 操作人 |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

### 9.2 热更新实现

后端可采用以下方式实现配置热更新：

1. **内存缓存**：将配置加载到内存中，修改时直接更新内存缓存。
2. **配置监听器**：实现配置变更监听器，当配置发生变化时自动重新加载。
3. **信号通知**：使用信号机制通知各服务组件重新加载配置。
4. **分布式配置**：如果是分布式系统，可使用分布式配置中心（如Nacos、Consul等）。

配置管理热更新接口设计遵循RESTful API规范，提供了完整的配置管理功能，包括获取配置、应用变更和恢复默认值等操作。通过这些接口，前端可以实现配置的实时管理和热更新，无需重启服务即可使配置生效，提高了系统的可靠性和可维护性。

同时，接口设计考虑了安全性、性能和可扩展性，为配置管理系统提供了坚实的后端支持。前端和后端的实现参考代码可以帮助开发者快速理解和实现配置管理功能，加速系统开发和部署。
